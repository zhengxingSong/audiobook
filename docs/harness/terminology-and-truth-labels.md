# Terminology And Truth Labels

> Purpose: prevent language drift.
> Rule: completion claims must use these labels consistently in code comments, reports, tests, and summaries.

## Required Labels

### `real`

Use `real` only when:

- the run used the actual GPT-SoVITS backend,
- the backend was actually contacted,
- the output file came from that path,
- the result is eligible for product acceptance.

### `surrogate`

Use `surrogate` when:

- audio was generated by any substitute path,
- the result exists only to validate infrastructure or UI flow,
- the run cannot be counted as product acceptance.

Examples:
- `demo://tone`
- mocked HTTP synthesis
- generated tone files
- placeholder playback assets

### `infrastructure-only`

Use `infrastructure-only` for runs that validate:

- web routing,
- job orchestration,
- report download,
- result download,
- basic artifact handling,

but do not prove the product claim.

### `product-acceptance`

Use `product-acceptance` only when:

- the real backend is used,
- the acceptance corpus is used,
- the acceptance rubric is applied,
- narration and character separation are evaluated from the produced audio,
- emotion rendering is evaluated from the produced audio.

### `temporary bridge`

Use `temporary bridge` for any implementation that is knowingly incomplete but retained to support progress.

Every temporary bridge must have:

- a unique name,
- an allowed scope,
- a reason for existence,
- a removal condition,
- an owner or responsible task,
- an entry in `temporary-bridges-register.md`.

## Forbidden Language

Do not use these words unless the product contract is actually satisfied:

- complete
- finished
- production-ready
- real conversion
- true audiobook generation
- GPT-SoVITS integrated

Do not use these phrases to blur the truth boundary:

- "basically works"
- "close enough to MVP"
- "realistic enough"
- "same as real backend for now"

## Required Summary Format

Any meaningful milestone summary must state:

- runtime mode: `real` or `surrogate`
- acceptance class: `infrastructure-only` or `product-acceptance`
- backend used
- whether narration and characters were separable
- whether emotion was audible or metadata-only

## Test Naming Rule

Tests must reflect truth labels in their names or comments.

Examples:

- `test_demo_endpoint_supports_infrastructure_flow`
- `test_real_gpt_sovits_run_passes_corpus_a`

Avoid ambiguous names like:

- `test_conversion_success`
- `test_full_app_ready`
