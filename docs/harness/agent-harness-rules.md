# Agent Harness Rules

## Operating Rules
- Treat these documents as the source of truth for execution scope.
- Make one focused change per iteration.
- Verify mechanically before claiming improvement.
- Log completed work before starting the next experiment.
- Prefer honest failure over disguised success.

## Forbidden Moves
- Do not use a demo or mock synthesis path as if it were the real product path.
- Do not claim the app is complete when narration, character separation, or emotion still fail.
- Do not widen scope because a nearby module looks convenient.
- Do not modify unrelated files to make the current task feel easier.
- Do not revert user changes or prior work outside your owned scope.

## Required Discipline
- State whether a change is a real product improvement or a temporary bridge.
- If a bridge exists, label it clearly and define the condition for removal.
- Keep reports aligned with observable outputs, not intentions.
- If the current phase cannot satisfy the product contract, stop and report the blocker instead of changing the definition of success.

## Completion Standard
- A task is complete only when the result is reproducible, test-backed, and consistent with the product contract.
